# Internationalisation (EN / हिन्दी)

Blueprint §20.1 requires the customer app to be *"bilingual-ready — EN + Hindi
strings via i18n keys from day one; Devanagari-safe fonts."* This is how that is
implemented, and — more importantly — where the line is drawn on what gets
translated.

## What must never be machine-translated

**Only UI chrome is in the message catalogs.** Chrome means navigation, button
labels, section headings, empty states, form labels, hints, and error banners —
copy authored by us in the TSX.

Everything below is deliberately **excluded** and renders in whatever language
the source data is in:

| Excluded | Why |
| --- | --- |
| Product name, brand, composition / salt | Drug identity. A renamed salt is a dispensing error. |
| Dosage, directions for use, warnings, contraindications, side effects, storage | Patient safety. Nobody on this project is qualified to translate clinical copy, and an LLM certainly isn't. |
| Prescription text, pharmacist notes, Schedule H1 register fields | Statutory records under the Drugs & Cosmetics Rules. They are kept verbatim. |
| Order numbers, addresses, customer/patient names, prices | Transactional identity — translating them breaks matching. |
| Legal pages (`/legal`, `/privacy`, `/terms`) | The English text is the operative version. A Hindi translation would need legal review before it could be shown as if authoritative. |

**Generic safety disclaimers are the exception and *are* translated.** "Information
is for reference; follow your doctor's advice", or the caution above the
substitutes rail, are not drug-specific clinical claims — they are instructions
to consult a professional. Leaving them as the one English line on an otherwise
Hindi screen makes them unreadable to exactly the person they protect, which is
strictly less safe than translating them. The rule is about *drug-specific
clinical data*, not about the word "doctor". Interpolated values inside such a
message (e.g. the composition string) still pass through verbatim.

If Hindi clinical copy is ever wanted, it has to come from the pharmacist as a
**separate authored field on the product record** (`descriptionHi`, etc.) with
its own review trail — not from this catalog, and not from a translation API.

## How it works

Routing-free `next-intl`: the locale lives in a cookie, not the URL.

- `src/i18n/config.ts` — locale list, default, cookie name, native labels.
- `src/i18n/request.ts` — resolves the locale per request; an unknown or
  tampered cookie value falls back to `en` rather than throwing.
- `src/i18n/types.ts` — augments `next-intl` so message keys are typechecked
  against `en.json`, **and** asserts `hi.json` covers every `en.json` key. Adding
  an English string without its Hindi counterpart fails `pnpm typecheck`.
- `src/messages/{en,hi}.json` — the catalogs.
- `src/app/actions/locale.ts` — server action that writes the cookie and
  revalidates the root layout.
- `src/components/LanguageToggle.tsx` — the picker, on Account → Preferences.

### Why a cookie and not `/hi/...`

This is an installed PWA for a single city, so language is a **user
preference**, not an SEO dimension. Locale-segmented routes would change
`start_url`, invalidate every existing deep link and push notification target,
and force a third route restructure after the `(app)` / `(marketing)` split —
all for search visibility we do not need in one market.

## Adding a string

1. Add the key to `src/messages/en.json`.
2. Add the same key to `src/messages/hi.json`. Typecheck enforces this.
3. Use it: `const t = useTranslations("shop"); t("myKey")`.

Two automated gates back this up:

- **`pnpm typecheck`** — keys must exist in `en.json` and must match between the
  two catalogs in both directions (`src/i18n/types.ts`).
- **`pnpm lint`** — runs `scripts/check-messages.mjs`, which parses every
  message as ICU and asserts en/hi agree on arguments and rich-text tags.
  Typecheck can only see key names; this sees *inside* the strings, where a
  dropped `{amount}` or a translated-away `<b>` would otherwise reach production
  in the language nobody on the team proof-reads.

Keep ICU placeholders (`{count}`, `{amount}`) identical between the two files.
Money and dates are formatted by the existing helpers and passed **in** as
placeholder values — never reformatted inside a message.

Hindi copy should read like an app, not like a textbook: the loanwords people
actually use (कार्ट, ऑर्डर, ऑफ़र, डिलीवरी) beat Sanskritised coinages.

## Not yet covered

Covered today: the app shell, shop, product detail, cart, checkout, orders list,
order detail, live tracking, and account.

These render in English regardless of locale. The infrastructure is in place;
they simply have no keys yet.

- The public marketing landing page (`(marketing)/`).
- Long-tail account screens: offers, referrals, refills, wishlist, profiles,
  prescriptions, notifications, notification settings.
- **Error boundaries** (`app/error.tsx`, `app/global-error.tsx`,
  `checkout/error.tsx`) — deliberate. `global-error.tsx` renders its own `<html>`
  and sits *above* the provider, so it has no translations available; and an
  error screen is the worst possible place to add a dependency that can itself
  throw. They stay English on purpose.
- `timeAgo()` in `src/lib/format.ts` emits English ("2m ago"). The tracking page
  uses next-intl's `useFormatter().relativeTime` instead, which is locale-correct
  via `Intl.RelativeTimeFormat`; the notifications page still uses the English
  helper and should switch when it is translated.
- The ops and driver apps — staff-facing, English-only by decision.
- Server-generated content: emails, SMS, push notification bodies, and the PDF
  invoice. These are rendered by the API, which has no locale plumbing; the
  user's language preference is not yet persisted server-side.

## A note on shared components

`OrderStatusBadge` is rendered by four screens. When it was translated, the
tracking page — which no one had touched — started showing a Hindi badge inside
English chrome. A shared component is a seam: translating one drags every
consumer into a half-translated state. When you translate a shared component,
grep for its consumers.

For the same reason, e2e specs must not select on translated copy. The status
badge exposes `data-testid="order-status"` and `data-value="<enum>"`; assert on
the enum, never the label.
